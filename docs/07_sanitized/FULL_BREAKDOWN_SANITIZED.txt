================================================================================
  SECTION 3: CUSTOM aPLib DECOMPRESSION ALGORITHM SPECIFICATION
================================================================================

  3.1 OVERVIEW

  The decompression algorithm is a CUSTOM variant of aPLib/aPack.
  Key differences from standard aPLib:
    - Uses sar eax,1 (signed shift) for offset encoding, not shr
    - LSB of offset shifted out determines COPY_SETUP path
    - Different gamma bit ordering
    - End-of-phase sentry: eax XOR 0xFFFFFFFF == 0
    - Phase 1 end triggers import resolution Phase 2

  3.2 ENTRY POINT DATA

  Decompressor:   RVA 0x3C4F30 (in .rsrc section)
  Source data:    RVA 0x355000 (raw offset 0x400 in file)
  Destination:    RVA 0x001000 (.text section start, 0x354000 bytes)
  Source reg:     rsi
  Destination reg: rdi

  3.3 BIT READER (GETBIT)

  Uses carry-flag trick via add + adc for bit extraction:
    add ebx, ebx    ; shift ebx left, CF = old MSB
    adc ebx, ebx    ; ebx = ebx*2 + CF (propagates through refill)

  ebx holds up to 31 bits. When exhausted (after 31 GETBIT calls):
    1. ebx refilled from 4 bytes at source (mov ebx, [rsi])
    2. adc at beginning of loop propagates old carry flag
    3. Double-pump technique means proper bit extraction per byte

  3.4 MAIN DECOMPRESSION LOOP

  1. Preload: mov dl, [rsi] (preload next byte)

  2. GETBIT -> CF
     CF=1: LITERAL BYTE
       - inc rsi, mov [rdi], dl, inc rdi
       - Preload next byte, goto step 1
     CF=0: MATCH (length,distance pair)

  3. OFFSET GAMMA DECODE:
     eax = prev_len + 1 (carried from previous match)

     FIRST PASS (jmp to LENGTH_LOOP_ENTRY):
       read 1 bit -> eax = eax*2 + bit
       read 1 stop bit -> if 0: continue loop

     SUBSEQUENT PASSES (jae -> LENGTH_DEC):
       dec eax
       read 2 bits -> eax = eax*2 + bit (each)
       read 1 stop bit -> if 0: continue loop
     After loop: eax -= 3

     OFFSET CLASSIFICATION:
       if eax < 0 (unsigned): SHORT MATCH - reuses ebp (last_offset)
       if eax >= 0: LONG MATCH - explicit offset:
         eax = (eax << 8) | dl     (merge next byte)
         src++
         eax = eax XOR 0xFFFFFFFF  (NOT)
         if eax == 0: END OF PHASE 1 (import resolution triggers)
         sar eax, 1               (signed right shift)
         LSB -> COPY_SETUP selector:
           CF=0: COPY_SETUP_A path
           CF=1: Alternate path at 0x3C5024
         movsxd rbp, eax           (sign-extend to last_offset)

  4. LENGTH DECODE:
     COPY_SETUP_A:
       getbit -> ecx = ecx*2 + bit

     ALT PATH (0x3C5024):
       inc ecx, getbit -> if 1: goto COPY_SETUP_A
       else: gamma-decode bits into ecx, ecx += 2

     COPY_SETUP_B:
       if last_offset < -0x500: ecx += 3
       else: ecx += 2

     Final copy_len = ecx

  5. MATCH COPY:
     copy_src = dst + last_offset
     if copy_len <= 5 or last_offset > -4:
       byte-by-byte copy loop
     else:
       dword copy: sub ecx, 4; jae loop; add ecx, 4
       byte remainder (0-3 bytes)

  6. Goto step 1 (loop until eax XOR 0xFFFFFFFF == 0 sentry)

  3.5 PHASE TRANSITION

  When eax XOR 0xFFFFFFFF == 0:
    - Decompression terminates
    - Import resolution begins (IAT entries resolved via GetProcAddress)
    - Execution continues with fully functional .text section

  B8 ?? ?? ?? ??          mov eax, <SSN>

  6.4 SSN RESOLUTION METHOD

  1. Locate Windows system directory (GetSystemDirectoryW)
  2. Read ntdll.dll from DISK at %WINDIR%\System32\ntdll.dll
     (NOT via GetModuleHandle - avoids hooked copy)
  3. Parse ntdll.dll export table for target function RVAs
  4. For each target function (e.g., "NtProtectVirtualMemory"):
     a. Read first bytes of function code in the file
     b. Identify pattern:
       - B8 XX XX XX XX (mov eax, <SSN>) - older encoding
       - 4C 8B D1 B8 XX XX XX XX (mov r10, rcx; mov eax, <SSN>) - Win10 1809+
     c. Extract 32-bit SSN immediate
  5. Self-patch stub code with resolved SSN
  6. Log: "stub built at %p (syscall# 0x%X)"

  6.5 PROTECTION LAYER FALLBACK

  Three-tier system:
    1. syscall (direct - primary)
    2. ntdll_direct (through ntdll.dll export - fallback)
    3. VirtualProtect (Win32 API - last resort)
  Error: "[Features] ALL LAYERS FAILED for addr=%p size=%zu err=%lu"

  8.1 HOOK INSTALLATION SEQUENCE

  At DllMain (after decompression):
    1. Decompress .text (custom aPLib)
    2. Resolve imports (IAT + lazy stub resolution)
    3. Read ntdll.dll from disk, parse syscall SSNs
    4. Find game.dll base via GetModuleHandleW

    6. Install hooks via NtProtectVirtualMemory + memory write
    7. Verify hooks: read back patched bytes, compare
    8. If count < expected: "WARNING: Game may have updated ... %d/%d

    11. Enter render loop

  8.2 HOOK TYPE CLASSIFICATION

  Type 1: NOP PATCH (~35 patterns)
    Replaces target instructions with 0x90 to disable functionality.
    Examples: grenade count, stim decrement, turret overheat, unlock check,
    landing speed, perception multiply, hash table INSERT, resource freeze,
    stamina drain, fuse timer, killstreak timer/score, combo mismatch,
    round completion, arcade combo, launch code, arrow sequence,
    shield relay cooldown, hellbomb arm timer, hover limit,
    railgun charge, turret timer, turret expire, coordinate/pipe timers

  Type 2: CONDITIONAL JUMP INVERSION (~6 patterns)
    JE/JNE/JL/JG changed to unconditional JMP or forced branch.
    Examples: boundary bypass, stratagem count bypass, difficulty tier force

  Type 3: FUNCTION RETURN (~5 patterns)
    Function prologue replaced with RET (0xC3) or MOV EAX,val; RET.
    Examples: ragdoll skip, recoil return, weather return, aggro return

  Type 4: FUNCTION PROLOGUE MATCHING (~8 patterns)
    FN_* patterns match function entry points for version verification,
    detour targeting, and update detection.

  Type 5: POINTER RESOLUTION (~6 patterns)
    PTR_* patterns resolve internal pointers/offsets:
      PTR_A: Session pointer resolver
      PTR_C: Unlock armory pointer
      PTR_D: Session pointer v2/alternate

  Type 6: CODE PATCH (~13 patterns)
    Custom byte modifications beyond NOP/JMP:
      SC-LIMIT code cave in bcrypt.dll
      Mass Strat Drop array index patches
      Weapon editor hook sites
      Map hack / Hoverpack control / Takeoff force modifications

  11.1 OVERVIEW

  The SC (Super Credits) farming system captures legitimate Mission/end API
  calls during normal gameplay and replays them with modified mission IDs
  to repeatedly claim rewards. This bypasses the game economy by exploiting
  the server's inability to detect replayed requests when hash table
  tracking is disabled client-side.

  11.2 TARGET API

  Endpoint: POST https://[REDACTED_GAME_API]/api/Operation/Mission/end
  (Found at 19 locations in the binary strings)

  11.3 STATE MACHINE

  IDLE -> CHECK_SESSION -> MONITORING (10s ring buffer) ->
    BATCH_FIRING (9 calls x 500ms) -> COOLDOWN (58s) -> GOAL_CHECK ->
    (toggle SC<->Medal, loop to MONITORING or STOP)

  Batch types: "Include Medals" (alternating SC/Medal) or "Medals Only"

  11.4 REQUEST CONSTRUCTION (Capture-and-Replay)

  HTTP Headers (captured from real game requests):
    Content-Type: application/json
    Authorization: <bearer/jwt token>
    X-Session: <session identifier>
    X-Auth: <alternative token>
    X-Token: <verification token>
    X-Signature: <cryptographic signature>
    Cookie: <session cookies>
    User-Agent: Mozilla/5.0

  JSON Request Body Fields:
    missionId          string   UUID - PRIMARY KEY for MIDSWAP
    capturedWarTime    uint32   War timestamp at capture
    url                string   Encoded request URL
    serObjOrigAddr     uint64   Original game memory address
    md                 hex str  Mission data (serialized binary)
    serObj             hex str  Serialized mission result
    slotData           hex str  Player loadout/slot config
    entityDeep         hex str  Deep entity state snapshot
    entityDataDeep     hex str  Deep entity data snapshot
    ac                 int      Activity/action count
    oi                 uint     Operation/objective index
    gs                 hex str  Game state snapshot

  Response parsing: {"amount": <int>, "missionId": "<uuid>"}

  Result categories:
    Calls sent, Skipped (replay=duplicate rejection),
    Rewarded (non-zero amount), Empty (zero/missing amount),
    Recovered (VEH crash restore), Failed (unrecoverable error)

  11.5 MIDSWAP (Mission ID Swapping)

  Two swap modes:
    1. STR swap: Replace missionId text in JSON body string
       "missionId":"<old>" -> "missionId":"<new>"
    2. BIN swap: Replace missionId in hex-encoded binary blobs
       Used for: serObj, md, entityDeep fields

  11.6 HASH TABLE NOP MECHANISM (Critical Bypass)

  Pattern at game_live.dll / game_current.dll:
    cmp eax, 0x10          ; threshold check
    jb  below              ; skip if bucket < 16
    mov ecx, [r14+0x08]    ; hash table pointer
    mov r9d, 0xFFFFFFFF   ; size auto-detect
    mov r8d, 0x70          ; entry size = 112 bytes

  Two INSERT call sites NOP-patched (26-28 bytes each).
  Effect: Hash table NEVER updated with collected SC entities.
  Game client reports NO SC collected; server grants as legitimate.
  Server duplicate detection bypassed because tracking entries never written.

  11.7 REPLAY CAPTURE SYSTEM

  Capture file: C:\libertea_replay_cap.json

  Capture via libcurl CURLOPT_WRITEFUNCTION hook on Mission/end POSTs.
  Replay modes: Single, Auto (configurable interval, default 2 min),
  Burst (N replays staggered 5s apart), Burst Loop (auto-fires at interval)

  11.8 SIGNATURE / NONCE CAPTURE ("f2s7" Protocol)

  Custom HTTP body signing/encryption via chunk analysis:
    - "f2s7" protocol identifier
    - Nonce extraction: "got f2s7+nonce (%lu bytes)"
    - Body XOR length: "bodyXorLen=%lu"

    - BCryptHashData for nonce/signature verification
