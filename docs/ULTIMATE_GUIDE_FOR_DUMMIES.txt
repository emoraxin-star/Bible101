1.1 THE SETTING AND STORY
--------------------------
Helldivers 2 is a third-person cooperative shooter game released in February
2024 by Arrowhead Game Studios, a Swedish game developer. It's the sequel to
Helldivers (2015), which was a top-down twin-stick shooter.

The story: In the far future, humanity has united under a single government
called "Super Earth." Super Earth is a satirical fascist regime that uses
propaganda, militarism, and "managed democracy" to control its population.
You play as a Helldiver — an elite shock trooper who drops from orbit onto
alien planets to fight for "freedom" and "democracy."

The game is heavily satirical, inspired by the movie Starship Troopers (1997).
Everything from the propaganda broadcasts to the tutorial narrator drips with
dark humor about militarism and authoritarianism.

1.2 THE GAMEPLAY LOOP
----------------------
Gameplay in Helldivers 2 works like this:

  STEP 1: On your ship (the "Super Destroyer"), you select a planet, a
          difficulty level (1 through 10), and a mission type.
  STEP 2: You and up to 3 teammates (friends or random matchmaking) drop
          onto the planet in "hellpods" — single-person drop capsules.
  STEP 3: You navigate the map, completing objectives like:
          - Destroying enemy bases
          - Rescuing civilians
          - Activating terminals
          - Launching ICBMs
          - Eliminating high-value targets
          - Collecting geological survey data
  STEP 4: You fight two enemy factions:
          - TERMINIDS: Giant alien bugs (like Starship Troopers). Swarm
            tactics, melee-focused, spawning from hives.
          - AUTOMATONS: Killer robots. Ranged combat, tanks, dropships,
            more tactical.
  STEP 5: You collect resources during the mission:
          - SAMPLES (common, rare, super): Used to upgrade your ship modules.
            Must be extracted with you — if you die, you drop them.
          - SUPER CREDITS: Premium currency found at Points of Interest (POIs)
            on the map. Can also be bought with real money.
          - MEDALS: Earned from completing missions and personal orders.
            Spent in Warbonds (battle passes).
          - REQUISITION SLIPS: Basic in-game currency earned from missions.
  STEP 6: You call in the extraction shuttle and survive until it arrives
          (usually ~2 minutes of defending the extraction point).
  STEP 7: Back on the ship, you spend your rewards on:
          - STRATAGEMS: Offensive/defensive/support abilities you call from
            orbit (airstrikes, turrets, supply drops, vehicles). Unlocked
            with Requisition Slips.
          - SHIP MODULES: Permanent upgrades to your Super Destroyer (faster
            stratagem cooldowns, more ammo, etc.). Upgraded with Samples.
          - WARBONDS: Battle passes that unlock weapons, armor, grenades,
            boosters, emotes, and cosmetics. Purchased with Medals. Some
            Warbonds cost Super Credits to unlock initially.

  SECTION 2: WHAT IS A GAME CHEAT / TRAINER?
================================================================================

2.1 THE BASIC CONCEPT
----------------------
A "game cheat" or "trainer" is a program that modifies a game while it's
running to give the player advantages that aren't normally available.

Think of it this way: imagine you're playing a board game. The game has rules
written in a rulebook. A cheat is like reaching over and changing the
rulebook while the game is in progress:

  NORMAL RULE: "When you pass GO, collect $200."
  CHEAT RULE:  "When you pass GO, collect $2,000."

The game still FOLLOWS the rules — but the rules have been altered.

2.2 COMPUTER GAMES AND MEMORY
------------------------------
In a computer game, the "rulebook" is stored in your computer's RAM (Random
Access Memory). RAM is a giant grid of numbered slots, each holding a number.
Each slot has an ADDRESS (where it is) and a VALUE (what number is stored there).

  Address #1,048,576 contains: 100   (this might be your health)
  Address #2,097,152 contains: 37    (this might be your ammo)
  Address #3,145,728 contains: 0     (this might be your Super Credits)
  Address #4,194,304 contains: "run" (this might be game code — instructions)

A cheat program can:
  a) FIND these addresses (where does the game store health?).
  b) READ the current values (what's my health right now?).
  c) WRITE new values (now make my health 99999).
  d) MODIFY the code instructions (change "subtract 1" to "subtract 0").

2.3 EXTERNAL VS INTERNAL CHEATS
--------------------------------
There are two main types of cheats:

  EXTERNAL CHEAT:
    - A SEPARATE program that runs alongside the game.
    - It uses Windows API functions like ReadProcessMemory() and
      WriteProcessMemory() to peek and poke at the game's memory from OUTSIDE.
    - Analogy: Looking through the window of a building and using a long stick
      to move things around inside.
    - PROS: Easier to write, doesn't require injection, less likely to crash
      the game.
    - CONS: Slower (context switches between programs), fewer capabilities,
      MUCH easier for anti-cheat to detect (reading/writing cross-process
      memory is a huge red flag).

  INTERNAL CHEAT:
    - Code that is INJECTED INSIDE the game process. It runs AS PART OF the
      game, sharing the same memory space.
    - Analogy: Sneaking inside the building and rearranging things from within.
    - PROS: Full access to everything (can read/write any memory instantly,
      call game functions directly, hook game code), harder to detect from
      outside (no cross-process activity).
    - CONS: Harder to write (must be a DLL, must handle injection, must not
      crash the game), any bug = instant game crash, must bypass anti-cheat
      that's running inside the same process.
    - LiberTea is an INTERNAL cheat.

2.4 REAL-WORLD ANALOGIES
-------------------------
  - A cheat is like a MOD for a game, except instead of adding new content,
    it changes the rules of the existing content.
  - An external cheat is like using a universal remote to control a TV from
    outside the room.
  - An internal cheat is like replacing the TV's circuit board with one that
    has extra features built in.
  - A "hook" is like a detour sign on a road — traffic that was going to
    Function A is rerouted to Function B instead.
  - A "pattern scan" is like finding a specific page in a book by searching
    for a unique phrase, rather than by page number.

3.4 HOW IT'S PROTECTED — ANTI-DETECTION MEASURES
--------------------------------------------------
LiberTea goes to extraordinary lengths to avoid being detected:

  LAYER 1 — PACKING (OBFUSCATION):
    The cheat DLL is compressed with a CUSTOM aPLib variant. The algorithm is
    NOT the standard aPLib — it uses bit inversion, modified gamma coding, and
    a unique offset calculation. This means:
    - Antivirus scanners can't match known signatures.
    - Manual reverse engineering requires figuring out the custom algorithm.
    - The packed DLL looks like random noise bytes.

  LAYER 2 — SYSCALLS (BYPASSING MONITORING):
    Normally, programs call Windows functions through DLLs (kernel32.dll,
    ntdll.dll). Anti-cheat tools can MONITOR these DLLs to detect suspicious
    activity. LiberTea SKIPS THE DLLs entirely and makes raw "syscall"
    instructions directly to the Windows kernel. This bypasses any monitoring
    that happens at the DLL level.

  LAYER 3 — GAMEGUARD BYPASS:
    Helldivers 2 uses nProtect GameGuard, a kernel-level anti-cheat driver.
    LiberTea blocks GameGuard's driver files (GameMon.des, GameMon64.des)
    before GameGuard can even start. It also uses SeDebugPrivilege elevation
    to gain the permissions needed to modify other processes.

  LAYER 4 — CRASH RECOVERY (VEH):
    If the cheat's hooks cause the game to crash, a Vectored Exception
    Handler (VEH) catches the crash, restores the original game code,
    logs what happened, and prevents the game from closing. This makes
    the cheat resilient to game updates that might break hooks.

  LAYER 5 — ANTI-DEBUG:
    The cheat checks if someone is analyzing it:
    - IsDebuggerPresent() — Windows API check for attached debuggers.
    - CheckRemoteDebuggerPresent() — Check for remote debugging.
    - RDTSC timing checks (34 occurrences) — Measure how long operations
      take; debuggers slow things down, so timing anomalies reveal analysis.
    - Process scanning for analysis tools.

  LAYER 6 — PROXY DETECTION:
    LiberTea checks for network monitoring tools:
    - Fiddler (web debugging proxy)
    - Burp Suite (security testing tool)
    - Charles Proxy (HTTP monitor)
    - mitmproxy (man-in-the-middle proxy)
    - 8+ other proxy/debugging tools
    If it detects any of these, it knows someone is watching its network
    traffic and can change behavior.

  LAYER 7 — HWID FINGERPRINTING:
    Each copy of the cheat is bound to a specific computer's hardware ID
    (derived from the MachineGuid registry key). This prevents sharing
    and allows the authors to ban specific users.

  LAYER 8 — ENCRYPTED COMMUNICATION:
    Communication with the cheat's backend servers uses:
    - Base64 encoding for encoding binary data as text.
    - bcrypt (Windows crypto API) for AES-CBC encryption and SHA256 hashing.
    - A custom "f2s7" protocol for HTTP body encryption with nonce-based

4.1 THE STICKY NOTE ON THE RULEBOOK
-------------------------------------
The simplest way to understand LiberTea:

  Imagine Helldivers 2 is a board game with a rulebook.
  The rulebook says: "When you get hit, lose 1 health."

  LiberTea puts a STICKY NOTE over that rule:
  "When you get hit, lose 0 health."

  The game still READS the rulebook. It still FOLLOWS the instructions it
  finds there. But the instructions have been replaced with different ones.

  That's it. That's the entire principle.

4.2 FINDING THE RULES
----------------------
But to put a sticky note on the right rule, you first need to FIND it.
The game's rulebook (its code, stored in RAM) is millions of lines long.

LiberTea finds specific rules using "pattern scanning." It searches for
UNIQUE byte sequences that identify a particular rule:

  Example: The health-damage function might always start with bytes:
    "48 89 5C 24 08 48 89 74 24 10"

  No other function in the game starts with this exact sequence. So when
  LiberTea scans the game's memory and finds this pattern, it knows:
  "AHA! This is the health-damage function. I'll put my sticky note HERE."

LiberTea has 73 of these patterns stored for different game functions.

4.3 CHANGING THE RULES
-----------------------
Once a rule is found, LiberTea has several ways to change it:

  NOP_PATCH: Erase an instruction entirely.
    Original: "subtract 10 from health"
    After:    "do nothing    do nothing do nothing"
    (The NOP instruction literally means "No Operation" — do nothing.)

  CODE_PATCH: Replace an instruction with a different one.
    Original: "set speed to 1.0"
    After:    "set speed to 5.0"

  FUNCTION_PROLOGUE: Redirect the start of a function.
    Original: "function doDamage() { calculate...; apply...; }"
    After:    "function doDamage() { return immediately; }"
    (The function is still called, but it immediately returns without doing
    anything. This is how God Mode works.)

  POINTER_RESOLVE: Change where a pointer points.
    Original: "player weapon -> rifle_01"
    After:    "player weapon -> super_gun_99"

  FUNCTION_RETURN: Modify what a function returns.
    Original: "isPlayerDead() returns TRUE"
    After:    "isPlayerDead() returns FALSE"

  CONDITIONAL_INVERT: Flip a yes/no check.
    Original: "IF player is reloading THEN prevent shooting"
    After:    "IF player is reloading THEN allow shooting" (JE -> JMP)

4.4 THE SERVER-SIDE CHALLENGE
-------------------------------
Some things (like health and ammo) are handled entirely on YOUR computer.
Changing them is straightforward — just modify the local memory.

But OTHER things (like Super Credits and Medals) are stored on the GAME
SERVER, not your computer. You can't just change a number in your RAM to
get 99999 Super Credits — the server doesn't care what your computer says.

This is where LiberTea's most clever trick comes in: REPLAY ATTACKS.

Instead of modifying a number, LiberTea:
  1. Watches your network traffic while you play legitimately.
  2. Records a "successful mission completion" message as it's sent to
     the server (like recording a phone call).
  3. Later, it plays that recording BACK to the server, over and over.

The server receives the recording and thinks: "Oh, the player just completed
another mission. Here are their rewards." It doesn't know the mission was
completed hours ago and is just being replayed.

This is like recording yourself saying "I'd like to withdraw $100" at a bank,
then playing that recording a hundred times. The bank teller keeps handing
you money because they hear the same authorized request.

To prevent the server from saying "wait, I've already seen this mission":
  - LiberTea disables the server's duplicate-checking code (the "hash table
    NOP" — two INSERT operations in the game's duplicate detection system
    are erased).
  - It changes the mission ID in each replay so they all look different
    (MIDSWAP — mission ID swapping).

5.1 THE INJECTOR (LIBERTEA_Bypass.exe)
---------------------------------------
What it is: A small .exe program you run before Helldivers 2.

What it does in simple terms: It's the "delivery truck" that transports the
cheat code into the game.

Step-by-step:

  STEP 1 — ASK FOR PERMISSION:
    Normally, programs on Windows can't see or touch each other's memory.
    This is like living in an apartment building where you can't enter your
    neighbor's unit. The injector gives itself a special key called
    "SeDebugPrivilege" that lets it access ANY program's memory.

  STEP 2 — DISABLE THE ALARM SYSTEM:
    GameGuard's driver files (GameMon.des and GameMon64.des) are like

    files so GameGuard never starts — the cameras never turn on.

  STEP 3 — DELIVER THE PACKAGE:
    The injector uses Windows functions to:
    a) Open the Helldivers 2 process (OpenProcess).
    b) Allocate some memory inside it (VirtualAllocEx) — like renting a
       storage unit inside the game's warehouse.
    c) Copy the cheat DLL into that memory (WriteProcessMemory).
    d) Create a thread inside the game that runs the DLL's startup code
       (CreateRemoteThread) — like hiring a worker inside the warehouse
       to unpack the delivery.
    e) The DLL's DllMain runs, and the cheat "wakes up" inside the game.

5.2 THE DLL (LIBERTEA.DLL)
----------------------------
What it is: The actual cheat code — a .dll file (like a .exe but it can't
run on its own — it needs a host process to load it).

What it does: Everything. Once injected, it goes through FOUR PHASES:

  PHASE 1 — UNPACK:
    The DLL arrives compressed (squished to hide from antivirus). The first
    thing it does is DECOMPRESS itself. This is like opening a vacuum-sealed
    bag — the contents expand to their full size.
    - Compressed size: 458 KB
    - Uncompressed size: 3.49 MB
    - The decompression code is a CUSTOM algorithm (modified aPLib).
    - This phase runs automatically when the DLL loads (within DllMain).

  PHASE 2 — RESOLVE IMPORTS (IAT FIXUP):
    The DLL needs to call Windows functions (like "draw a window" or "send
    network data"). It doesn't know where these functions live until it
    builds an "address book" — the Import Address Table (IAT).
    - The packed DLL's IAT is scrambled (intentionally, to confuse analysis).
    - Phase 2 unscrambles it and fills in the correct addresses.
    - After this, the DLL can call any Windows function it needs.

  PHASE 3 — FIND AND PATCH THE GAME:
    Now the cheat scans the game's code for specific patterns (73 of them).
    Each pattern is like a fingerprint that identifies a specific game
    function. When a pattern is found, the cheat installs a HOOK at that

  PHASE 4 — CREATE THE OVERLAY:

5.3 THE PACKER (aPLib Compression)
-----------------------------------
What it is: The system that compresses the cheat DLL to hide it.

Why compression matters:
  1. STEALTH: Anti-virus and anti-cheat programs scan files for known cheat
     patterns. A compressed file looks like random noise — the scanners can't
     recognize it.
  2. SIZE: The unpacked .text is 3.49 MB. The packed version is 458 KB —
     about 1/8th the size. Smaller files are easier to inject and harder to
     notice.
  3. PROTECTION: You can't analyze the code while it's compressed. You have
     to figure out the unpacking algorithm first (which is custom).

The aPLib algorithm (explained simply):
  aPLib is a compression algorithm by Joergen Ibsen. The standard version
  works like this:
    - Read a compressed bitstream.
    - Each bit indicates: "literal byte" (copy this byte directly) or
      "match" (go back X bytes and copy Y bytes from the already-
      decompressed output).
    - Matches save space because you reference data you've already
      decompressed.

  LiberTea's CUSTOM version modifies three things:
    1. BIT INVERSION: The bit reader logic is inverted (add/adc vs shl/rcl).
    2. GAMMA CODING: The length encoding (how many bytes to copy) uses a
       modified Elias gamma code with decrement-then-read logic.
    3. OFFSET CALCULATION: The "go back X bytes" part uses XOR with
       0xFFFFFFFF and arithmetic right shift instead of standard calculation.

  The effect: Even if you KNOW it's aPLib, you can't decompress it with
  standard aPLib tools. You must reverse engineer the modifications.

  How the unpacking happens inside the DLL:
    - The DLL's entry point (DllMain), at RVA 0x3C4F30, contains a small
      assembly-language unpacker stub.
    - This stub is only about 200 bytes of pure x64 assembly.
    - It reads from RVA 0x355000 (the compressed data, at raw file offset
      0x400) and writes to RVA 0x001000 (the .text section base).
    - The stub was probably compiled separately (maybe in pure ASM or with a
      special compiler flag) to be position-independent and runtime-free.

5.4 THE OVERLAY (ImGui Menu)
-----------------------------
What it is: The cheat's user interface — a settings window with tabs,
checkboxes, sliders, and buttons that appears on top of the game.

How it renders:
  1. The overlay is a transparent WINDOW (created with Win32 API) that sits

5.5 THE SYSCALL SYSTEM (Direct Kernel Calls)
----------------------------------------------
What it is: A way to bypass Windows security monitoring.

Normal flow (what most programs do):
  Program -> kernel32.dll -> ntdll.dll -> Kernel

LiberTea's flow:
  Program -> direct syscall -> Kernel

Why this matters: Anti-cheat programs HOOK ntdll.dll functions. They put
their own monitoring code at the beginning of important functions like
NtProtectVirtualMemory. When a normal program calls NtProtectVirtualMemory,
the anti-cheat sees it and can decide if it's suspicious.

LiberTea SKIPS ntdll.dll entirely. It makes a raw "syscall" instruction,
which goes directly to the kernel. The anti-cheat never sees it.

How syscalls work in LiberTea:
  - Each Windows syscall has a "System Service Number" (SSN) — like a phone
    extension.
  - Example: NtProtectVirtualMemory might be SSN 0x50 on Windows 10 and
    SSN 0x4F on Windows 11. The numbers change between Windows versions.
  - LiberTea dynamically reads ntdll.dll at runtime to find the correct SSN
    for the current Windows version.
  - It constructs a small assembly stub for each syscall:
      lea rcx, [function_name]    ; Load the function name
      load SSN into eax           ; Get the system service number
      mov eax, SSN                ; Set the syscall number
      syscall                     ; Make the call
  - LiberTea has 9 of these syscall stubs for different kernel functions.

This technique is used by both sophisticated cheats and sophisticated malware
for the same reason: it's hard to detect without kernel-level monitoring.

  The C++ compiler from Microsoft. The go-to tool for building Windows
  programs. LiberTea was compiled with MSVC 2022.

MUTEX:
  A lock that prevents two threads from accessing the same resource
  simultaneously. Like a token that says "only the person holding this
  token can use the printer right now."

--- N ---

NOP (No Operation):
  An instruction that does nothing. Bytes: 0x90 (x86) or 0x90 0x90 (x64
  if 2-byte alignment needed). Used to "erase" existing instructions.

NTDLL.DLL:
  The lowest-level Windows DLL. All system calls eventually pass through
  ntdll. LiberTea READS ntdll to find syscall numbers but SKIPS calling
  through it, instead making syscalls directly.

--- O ---

OFFSET:
  A distance from a reference point. "The health value is at offset +0x2C0
  from the player object." Like saying "third house on the right" instead
  of giving the full street address.

OPCODE:
  The numeric code for a CPU instruction. 0x90 = NOP, 0xC3 = RET (return),
  0xE9 = JMP (jump). Like the barcode for a specific instruction.

OVERLAY:
  A transparent window rendered on top of another application. LiberTea's
  menu is an overlay on top of Helldivers 2. Like drawing on a clear
  plastic sheet placed over a photograph.

--- P ---

PACKER:
  Software that compresses an executable and adds unpacking code. The file
  is compressed on disk and decompresses when run. Like shrink-wrapping a
  product with a tear strip for easy opening.

PATTERN / SIGNATURE:
  A unique sequence of bytes used to find specific code in memory. Like a
  fingerprint: no two patterns match the same function. Wildcard bytes (??)
  mean "match anything here."

PE (Portable Executable):
  The file format for Windows executables (.exe) and DLLs (.dll). Like the
  standard box format for shipping products — everything has the same
  structure.

PROCESS:
  A running program. When you double-click an .exe, Windows creates a
  process — a container with memory, threads, and resources.

PROLOGUE:
  The beginning of a function. Usually sets up the stack frame and saves
  registers. Like the opening credits of a movie.

--- R ---

RDTSC (Read Time-Stamp Counter):
  A CPU instruction that reads a high-precision timer. LiberTea uses it to
  detect debuggers: if operations take too long (because a debugger is
  stepping through them), the cheat knows it's being analyzed.

REGISTER:
  A tiny, ultra-fast storage location inside the CPU. The CPU has ~16
  general-purpose registers (RAX, RBX, RCX, RDX, etc.). Like the CPU's
  pocket — fastest access but very limited space.

REPLAY ATTACK:
  Recording legitimate data and replaying it later. LiberTea records a
  mission completion and replays it to farm Super Credits.

RTTI (Run-Time Type Information):
  Metadata that C++ programs include so they know class types at runtime.
  Like name tags that objects wear so the program knows what they are.

RVA (Relative Virtual Address):
  An address expressed as an offset from the start of the file in memory.
  Like saying "page 50" instead of "the 50th page from the beginning of
  the book."

--- S ---

SECTION (.text, .data, .rdata, etc.):
  A PE file is divided into sections with different purposes and
  permissions. .text = code, .data = variables, .rdata = constants.
  Like rooms in a house: kitchen (code), bedroom (data), library (constants).

SEOEBUG PRIVILEGE:
  A Windows privilege that allows a program to debug (and therefore access)
  other programs. Like a master key for all apartments. The injector enables
  this.

SEH (Structured Exception Handling):
  Windows' standard crash-handling system. Uses try/catch blocks. LiberTea
  has 2,466 SEH entries (.pdata records).

SHA256:
  A cryptographic hash function. Takes any input and produces a 256-bit
  (32-byte) "fingerprint." Like a super-precise fingerprint for data.

SIGNATURE:
  See PATTERN.

SSN (System Service Number):
  The unique number identifying each Windows syscall. Like a phone extension:
  "NtProtectVirtualMemory? That's extension 0x50."

STACK:
  A region of memory for function calls and local variables. Last-in,
  first-out. Like a stack of plates: you add plates to the top and take
  them from the top.

STATIC CRT:
  The C Runtime Library compiled INTO the executable, not loaded from an
  external DLL. Makes the file bigger but eliminates dependencies.

STRING:
  A sequence of text characters. In C/C++, usually null-terminated (ends
  with a 0 byte). Programs use strings for messages, names, URLs, etc.

SUBCLASSING:
  A Windows technique where you intercept messages sent to a window. Like
  redirecting someone's mail to your address so you can read it first.

SYSCALL:
  The instruction that transitions from user mode to kernel mode. The most
  direct way to ask the OS to do something. Like a hotline directly to the
  building manager, bypassing the front desk.

--- T ---

THREAD:
  A sequence of instructions running independently. A process can have
  many threads running simultaneously. Like having multiple workers in a
  kitchen, each doing different tasks.

TLS CALLBACK:
  A function that runs BEFORE the main entry point. Used by some packers
  and anti-debug code. LiberTea does NOT use TLS callbacks.

TRAMPOLINE:
  A small piece of code that bounces execution from a hook back to the
  original function. Like a trampoline in gymnastics: you jump to the
  trampoline, which bounces you to where you wanted to go.

--- V ---

VEH (Vectored Exception Handler):
  A crash catcher. Unlike SEH (which is function-scoped), VEH is process-wide.
  LiberTea registers a VEH to catch crashes from its hooks and restore the
  original code. Like a safety net under a trapeze act.

VTABLE (Virtual Function Table):
  A table of function pointers used by C++ classes with virtual methods.
  Like a phone directory for an object's methods.

--- W ---

WILDCARD BYTE (??):
  In a pattern signature, "??" means "match any byte here." Used when a
  byte changes between game versions. Like "Find the person wearing a ???
  colored hat" — the hat color doesn't matter.

WINHTTP / WININET:
  Windows libraries for making HTTP requests. Part of LiberTea's triple-
  stack HTTP system.

WORKERS (Cloudflare Workers):
  A serverless computing platform. LiberTea uses workers.dev for its CDN
  and update distribution.

--- X ---

XOR (Exclusive OR):
  A simple reversible bit operation. A XOR B = C, and C XOR B = A. Used
  in LiberTea's f2s7 protocol for body obfuscation. Like a light switch
  where flipping it twice returns to the original state.

  SECTION 8: THE GAMEGUARD BYPASS EXPLAINED
================================================================================

8.1 WHAT IS GAMEGUARD?
-----------------------
nProtect GameGuard is a kernel-level anti-cheat system developed by INCA
Internet, a South Korean security company. It's used by many Asian MMOs and,
as of 2024, Helldivers 2.

GameGuard installs a KERNEL DRIVER (GameMon.des on 32-bit, GameMon64.des on
64-bit) that runs at the deepest level of your operating system. From this
privileged position, it can:
  - Monitor ALL memory access.
  - Detect debuggers and analysis tools.
  - Block suspicious programs from running.
  - Scan for known cheat signatures.
  - Prevent process memory from being read or modified.

Think of GameGuard as a security guard who has the keys to EVERY room in
the building, including the maintenance areas and security office.

8.2 HOW GAMEGUARD DETECTS CHEATS
----------------------------------
GameGuard uses multiple detection methods:

  1. SIGNATURE SCANNING: Scans running processes' memory for known cheat
     patterns (byte sequences that match known cheat DLLs).

  2. BEHAVIORAL DETECTION: Monitors for suspicious actions:
     - Other programs opening handles to the game process.
     - Memory page protections being changed (NtProtectVirtualMemory calls).
     - New threads being created in the game process.
     - DLLs being loaded that shouldn't be there.

  3. INTEGRITY CHECKING: Verifies that game files and memory haven't been
     modified. Compares checksums of game code against known-good values.

  4. DEBUGGER DETECTION: Detects debuggers attached to the game process.

  5. DRIVER BLOCKING: Prevents other kernel drivers from interfering with
     GameGuard's own driver.

  6. HOOK DETECTION: Checks if Windows API functions have been modified
     (hooked) to hide activity.

8.3 HOW LIBERTEA BYPASSES GAMEGUARD
-------------------------------------
LiberTea's bypass is clever and multi-layered:

  STEP 1 — BLOCK THE DRIVERS AT BOOT:
    GameGuard's files (GameMon.des, GameMon64.des) must exist on disk for
    GameGuard to load. LiberTea's injector:
    a) Deletes these files if they exist.
    b) Creates dummy/blocked files in their place to prevent GameGuard from
       recreating them.
    c) Sets file permissions to deny GameGuard's installer access.
    This is like removing the security guard's uniform and badge BEFORE
    their shift starts — they can't work without them.

  STEP 2 — ELEVATE PRIVILEGES:
    The injector gives itself SeDebugPrivilege, which provides the same
    level of access a debugger would have. This lets it:
    a) Open a handle to the game process.
    b) Allocate memory in the game's address space.
    c) Write data (the cheat DLL) into that memory.
    d) Create a thread to execute the DLL.

  STEP 3 — DIRECT SYSCALLS:
    Once injected, the cheat doesn't call Windows APIs through the normal
    DLL path. Instead, it makes direct syscall instructions. This bypasses
    GameGuard's user-mode hooks on ntdll.dll.

  STEP 4 — VEH CRASH RECOVERY:
    If GameGuard (or the game itself) causes a crash due to the hooks,
    the VEH catches it, restores original code, and prevents detection.

8.4 WHY KERNEL-LEVEL MATTERS
------------------------------
There is an "arms race" between cheats and anti-cheats:

  User-mode (application level):
    - Easy to develop and deploy.
    - Easy to detect (other user-mode programs can see what you're doing).
    - LiberTea operates here mainly, but uses syscalls to touch kernel space.

  Kernel-mode (driver level):
    - Very hard to develop (requires kernel programming expertise, driver
      signing certificates).
    - Very hard to detect (only other kernel drivers can monitor you).
    - GameGuard operates here.
    - LiberTea doesn't have its own driver, but it bypasses GameGuard's
      driver by preventing it from loading.

The fact that LiberTea can defeat a kernel-level anti-cheat without
installing its own kernel driver (which would be extremely risky and
detectable) shows sophisticated design.

A: The overlay uses Dear ImGui + OpenGL:
   1. Creates a transparent window on top of the game window.
   2. Hooks wglSwapIntervalEXT (called every frame during buffer swap).
   3. During the swap, injects ImGui rendering commands.
   4. Uses window subclassing + WM_SC_DISPATCH for thread-safe rendering.
   5. Stays invisible until you press INSERT/HOME.
   The result looks like a native in-game settings menu.

A: f2s7 is a custom protocol for encrypted communication between the cheat
   and its backend servers. Features:
   - Nonce-based session initialization (unique number used once).
   - Body XOR encryption (simple but hides content from casual inspection).
   - Key-waiting mechanism (waits for server to provide encryption key).
   - Used for license verification, update checking, and telemetry.
   Full spec in logs/agentG_f2s7_spec.txt.

Q10: HOW WAS THE .text SECTION UNPACKED?
A: We couldn't fully implement the custom aPLib algorithm in Python (partial

    LoadLibrary():

      # DllMain runs automatically -> decompresses .text -> resolves IAT
      ctypes.memmove(buf, handle + 0x1000, 0x354000)
      # buf now contains the fully unpacked .text section
    This dumped the ground-truth unpacked data to .text_unpacked_mem.bin.

================================================================================
  APPENDIX B: READING THE ANALYSIS DOCUMENTS
================================================================================

The analysis documents use consistent conventions:

  RVA (Relative Virtual Address):
    An address relative to the DLL's load base. For example, RVA 0x104DAD
    means "offset 0x104DAD from wherever the DLL is loaded."
    To get the real address: real_address = dll_base + RVA.

  HEXADECIMAL NUMBERS:
    Prefixed with "0x". 0x10 = 16 decimal, 0xFF = 255 decimal.
    Memory addresses and offsets are almost always shown in hex.

  "??" IN PATTERNS:
    A wildcard byte that matches anything. "48 89 ?? 24" matches 48 89 00 24,
    48 89 01 24, 48 89 FF 24, etc. Wildcards are used when a byte changes
    between game versions or is a variable operand.

  HOOK NOTATION:
    "NOP_PATCH at game.dll+0x1A2B3C: 48 89 5C 24 -> 90 90 90 90"
    Means: "At offset 0x1A2B3C inside game.dll, the 4 original bytes were
    replaced with 4 NOP instructions."

  SYSCALL STUB NOTATION:
    "lea rcx, [name]; load SSN; mov eax, SSN; syscall"
    Shows the 4 steps to make a direct system call:
    1. Load the function name (for reference/identification).
    2. Load the system service number.
    3. Move SSN into eax (the register for syscall number).
    4. Execute the syscall instruction.

  FUNCTION TYPES:
    - FUNCTION_PROLOGUE: Hook at function START.
    - FUNCTION_RETURN: Hook at function END (epilogue).
    - CODE_PATCH: Replace instruction bytes anywhere.
    - NOP_PATCH: Fill bytes with NOPs (0x90).
    - POINTER_RESOLVE: Read/write a pointer value.
    - CONDITIONAL_INVERT: Flip a branch (JE -> JMP, JNE -> JMP).

================================================================================
  APPENDIX F: DEEP DIVE — THE FOUR INITIALIZATION PHASES
================================================================================

Understanding the 4-phase initialization is key to understanding how LiberTea
works. Let's walk through each phase in detail.

F.1 PHASE 1 — UNPACK (.text Decompression)
--------------------------------------------

When DllMain runs, the FIRST thing that happens is decompression. The DLL's
code section (.text) is stored in compressed form at a specific location in
the DLL file.

The unpacker stub is a small piece of assembly code (~200 bytes) located at
RVA 0x3C4F30. This stub:

  1. Sets up registers and a workspace.
  2. Reads from the compressed data source (RVA 0x355000).
  3. Writes decompressed output to the .text section (RVA 0x001000).
  4. Uses a bit-reading loop: add ebx,ebx / adc ebx,ebx.
     - The carry flag (CF) determines each "bit" from the compressed stream.
  5. For each bit:
     - If the next bit is 1 (CF=1): This is a LITERAL byte — copy it directly
       to the output.
     - If the next bit is 0 (CF=0): This is a MATCH — go back X bytes and
       copy Y bytes from already-decompressed data.
  6. The match offset uses a GAMMA coding scheme:
     - First, the length code is read using Elias gamma encoding.
     - Then, if it's a "long match," the offset is calculated as:
       eax = (eax << 8) | next_byte
       eax = eax XOR 0xFFFFFFFF
       eax = sar eax, 1 (arithmetic right shift)
     - The LSB (least significant bit) of this result determines whether
       to use the standard copy path or an alternative path.
  7. The copy loop uses dword (4-byte) copying for efficiency, with a
     remainder loop for the last 1-3 bytes.

Why this is complex: The authors modified the STANDARD aPLib in three ways:
  a) Bit reader uses add/adc instead of shl/rcl (bit inversion).
  b) Gamma decoding subtracts before reading instead of after.
  c) Offset calculation uses XOR 0xFFFFFFFF + sar instead of the standard
     calculation.

Result: Source 458 KB -> Destination 3.49 MB. Compression ratio: ~7.6:1.

F.2 PHASE 2 — RESOLVE IMPORTS (IAT Fixup)
-------------------------------------------

After the .text section is unpacked, the DLL has its code, but the code
contains "stubs" — placeholders where calls to external Windows functions
should go. These stubs are resolved by:

  1. Parsing the Import Directory in the PE header.
  2. For each imported DLL (kernel32.dll, user32.dll, ntdll.dll, etc.):
     a) Load the DLL with LoadLibrary (if not already loaded).
     b) For each imported function:
        - Find the function in the DLL with GetProcAddress.
        - Write the function's address into the IAT (Import Address Table).
  3. After this phase, function calls like:
       call [gap_where_kernel32_CreateFileW_should_be]
     become:
       call [0x7FFE12345678]  (real address of CreateFileW in kernel32)

The IAT resolution is necessary because the packed DLL's IAT is SCRAMBLED —
the original author intentionally broke it so that the unpacked DLL cannot
run correctly without the custom IAT fixup code in DllMain. This is both
an anti-reverse-engineering measure and a practical necessity (the packer
doesn't know the final memory layout of the unpacked code).

F.3 PHASE 3 — PATTERN SCAN AND HOOK INSTALL
---------------------------------------------

This is the CORE of the cheat. The DLL scans the game's modules for specific
byte patterns and installs hooks at those locations.

Step-by-step:

  1. FIND THE GAME BASE:
     - GetModuleHandle("game.dll") — finds where game.dll is loaded in memory.
     - If not found, waits and retries (the DLL is loaded after the cheat
       in some injection scenarios).

  2. ENUMERATE PATTERNS:
     - The cheat has a built-in database of 404 IDA-style patterns.
     - Each pattern has:
       * A name string (e.g., "GodMode_PlayerOnly_Hook").
       * A module name (which DLL to search).
       * A byte signature with wildcards.
       * A hook type (NOP_PATCH, CODE_PATCH, etc.).
       * The bytes to write (for CODE_PATCH) or an action (for other types).

  3. SCAN FOR EACH PATTERN:
     - For each pattern, scan the target module's memory (e.g., game.dll)
       for the byte sequence.
     - Wildcards (??) match any byte.
     - The scanner is a SCALAR comparison — it checks byte by byte. No SIMD
       (no SSE/AVX) — just a simple loop.
     - The scanner has a "light mode" option that only checks the first
       few non-wildcard bytes for an initial filter.

  4. VERIFY AND REPORT:
     - Count how many patterns were found.
     - If less than expected: "WARNING: Game may have updated ... X/Y
       patterns found."
     - This warns the user that the cheat may not be compatible with the
       current game version.

  5. INSTALL HOOKS:
     - For each found pattern, the cheat determines the hook type:
       * NOP_PATCH: Change memory protection to writable (VirtualProtect or
         NtProtectVirtualMemory via direct syscall), then write NOP bytes
         (0x90) over the original instructions.
       * CODE_PATCH: Write the specified replacement bytes.
       * FUNCTION_PROLOGUE: Write a JMP instruction at the function start
         that redirects to the cheat's handler. Save the original bytes
         for unhooking.
       * POINTER_RESOLVE: Read or write a pointer value at the offset.
       * FUNCTION_RETURN: Hook at the function's epilogue (return point).
       * CONDITIONAL_INVERT: Change a conditional jump (JE) to an
         unconditional jump (JMP) or vice versa.
     - Memory protection is changed via DIRECT SYSCALL to NtProtectVirtual-
       Memory, bypassing any monitoring on the regular VirtualProtect path.

  6. SYSCALL STUB CONSTRUCTION:
     - 9 syscall stubs are constructed dynamically.
     - For each needed syscall (like NtProtectVirtualMemory):
       a) Read ntdll.dll's export table to find the function.
       b) Read the first few bytes of the function — they contain the SSN
          (System Service Number) — e.g., "mov eax, 0x50; syscall".
       c) Parse out the SSN value.
       d) Store it for use in the custom stub.
     - The custom stub: lea rcx, [name]; mov eax, SSN; syscall.

  7. FORMAT A WARNING IF PATTERNS MISSING:
     - Shows a message box or ImGui warning if the expected number of
       patterns wasn't found.
     - This is how users know the cheat needs an update.

F.4 PHASE 4 — OVERLAY RENDERING
---------------------------------

Once the hooks are installed, the cheat creates the UI overlay:

  1. CREATE ImGui CONTEXT:
     - ImGui::CreateContext() initializes the ImGui system.
     - Sets up font, style, and rendering context.

  2. FIND THE GAME WINDOW:
     - FindWindow or EnumWindows locates the Helldivers 2 window.
     - Gets the window's OpenGL rendering context (HGLRC).

  3. INSTALL RENDERING HOOK:
     - wglSwapIntervalEXT is the function called every frame during the
       "buffer swap" — when the finished frame is sent to the display.
     - The cheat hooks this function by:
       a) Getting the function address with wglGetProcAddress.
       b) Saving the original bytes at that address.
       c) Writing a JMP to the cheat's rendering handler.

  4. RENDERING HANDLER:
     - Every frame, after the game draws, the cheat's handler runs:
       a) Start a new ImGui frame.
       b) If the menu is visible (toggle key pressed), render all ImGui
          widgets (checkboxes, sliders, buttons, text).
       c) End the ImGui frame and render to the OpenGL context.
       d) Then call the ORIGINAL wglSwapIntervalEXT to finish the swap.

  5. WINDOW SUBCLASSING:
     - The game's window procedure (WndProc) is subclassed — replaced with
       a custom WndProc that intercepts window messages.
     - A custom message (WM_SC_DISPATCH) is registered with RegisterWindow-
       Message.
     - When the rendering thread needs the main thread to do something, it
       posts WM_SC_DISPATCH.
     - This ensures thread-safe coordination between the game's main thread
       and the overlay's rendering thread.

  6. KEYBOARD HOOK:
     - A keyboard hook captures key presses (INSERT, HOME, etc.) to toggle
       the menu visibility and navigate between tabs.
     - ImGui handles the actual keyboard input processing.

     - Sets proper headers (Content-Type: application/json, User-Agent,
       Authorization tokens from the game session).

G.4 DUPLICATE PREVENTION BYPASS — THE HASH TABLE NOP
------------------------------------------------------
The server normally prevents duplicate mission submissions by storing
mission IDs in a hash table:

  When a Mission/end POST arrives:
    1. Check if missionId is in the hash table.
    2. If YES: reject (duplicate).
    3. If NO: INSERT missionId into the hash table and process the request.

LiberTea defeats this by finding the CODE that does the INSERT and NOP-ing it:

  Original code:
    mov rcx, [hash_table]     ; load hash table pointer
    lea rdx, [mission_id]     ; load mission ID
    call HashTable_Insert     ; INSERT into table

  Modified code (NOP_PATCH):
    nop                       ; (was mov rcx, ...)
    nop                       ; (was lea rdx, ...)
    nop                       ; (was call HashTable_Insert)
    (7+ bytes of NOPs)

  Result: The INSERT never happens. The hash table stays empty. Every
  replay is treated as a new mission.

  There are TWO INSERT sites in the code, both NOP'd.

G.5 MIDSWAP — MISSION ID ROTATION
-----------------------------------
Even with the hash table NOP'd, sending the SAME missionId repeatedly is
suspicious. LiberTea rotates the missionId:

  - The captured replay has missionId "A".
  - For batch 1: Replace "A" with random "B".
  - For batch 2: Replace "A" with random "C".
  - For batch 3: Replace "A" with random "D".
  - etc.

  The missionId format is a UUID/GUID: "xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx".
  LiberTea generates random GUIDs or uses a pool of pre-generated ones.

  1. The cheat identifies all player IDs in the squad.
  2. For each replay batch, it modifies the POST body to assign rewards

  3. Batch 1: Player A gets SC.
     Batch 2: Player B gets SC.
     Batch 3: Player C gets SC.
  4. This distributes SC across the entire squad, so everyone benefits.

G.7 VEH CRASH RECOVERY — KEEPING THE LOOP RUNNING
---------------------------------------------------
SC farming can run for HOURS unattended. If a game update or network error
causes the cheat to crash, the VEH catches it:

  1. The Vectored Exception Handler is registered with:
     AddVectoredExceptionHandler(1, CrashHandler)
     (The "1" means this handler runs BEFORE standard SEH handlers).

  2. When a crash occurs:
     a) CrashHandler is called with EXCEPTION_POINTERS.
     b) The handler logs: BEFORE/AFTER crash data (exception code, address,

     c) The handler RESTORES the original game code (unhooks any hooks that
        may have been corrupted).
     d) The handler returns EXCEPTION_CONTINUE_EXECUTION — the program
        continues running from the crash point.
     e) If recovery fails, returns EXCEPTION_CONTINUE_SEARCH to let the
        normal crash handler take over (game might still crash, but at
        least the cheat tried).

  3. This makes the SC farming loop resilient to many types of crashes.

    - Medal value: enough to complete 2-3 Warbonds.

  These are theoretical maximums. Actual rates depend on:
    - Server rate limiting (if the server detects unusual patterns).
    - Network latency (each request takes real time).
    - The captured mission's reward structure.
    - Anti-cheat interventions.

================================================================================
  APPENDIX H: THE INJECTION PROCESS — COMPLETE TECHNICAL WALKTHROUGH
================================================================================

H.1 BEFORE INJECTION — PRIVILEGE SETUP
----------------------------------------
The injector (LIBERTEA_Bypass.exe) first elevates its privileges:

  1. OpenProcessToken(GetCurrentProcess(), TOKEN_ADJUST_PRIVILEGES, &hToken).
     - Gets a handle to the current process's security token.

  2. LookupPrivilegeValue(NULL, SE_DEBUG_NAME, &luid).
     - Looks up the Locally Unique Identifier (LUID) for SeDebugPrivilege.
     - SeDebugPrivilege is the "debug privilege" that allows accessing other
       processes' memory.

  3. AdjustTokenPrivileges(hToken, FALSE, &tp, 0, NULL, NULL).
     - Enables SeDebugPrivilege for this process.
     - Without this, OpenProcess with PROCESS_ALL_ACCESS on another process
       would fail.

H.2 GAMEGUARD DRIVER BLOCKING
-------------------------------
Before the game starts, the injector neutralizes GameGuard:

  1. Check if GameMon.des / GameMon64.des exist on disk.
     - Common locations: game directory, system32, drivers folder.

  2. If found, delete them or rename them.
     - DeleteFile or MoveFile to a different name/extension.

  3. Create dummy files in their place.
     - CreateFile with 0 bytes, set to read-only.
     - This prevents GameGuard's installer from recreating them.

  4. Optionally block the file paths.
     - Set file permissions to deny write access to the GameGuard installer.

  5. Check running drivers.
     - If GameMon is already loaded (from a previous game session), the
       injector may:
       a) Stop the driver service (sc stop GameMon).
       b) Delete the driver service (sc delete GameMon).
       c) Require a reboot to fully remove.

H.3 DLL INJECTION — THREE METHODS
-----------------------------------
The injector can use one of several injection methods:

  METHOD 1: CreateRemoteThread (CLASSIC)
    a) OpenProcess(PROCESS_ALL_ACCESS, FALSE, game_pid).
       - Opens the game process with full access rights.
    b) VirtualAllocEx(hGame, NULL, dll_path_size, MEM_COMMIT, PAGE_READWRITE).
       - Allocates memory inside the game's address space.
    c) WriteProcessMemory(hGame, remote_addr, dll_path, dll_path_size, NULL).
       - Writes the DLL path string into the allocated memory.
    d) GetProcAddress(GetModuleHandle("kernel32.dll"), "LoadLibraryW").
       - Gets the address of LoadLibraryW (the function that loads DLLs).
    e) CreateRemoteThread(hGame, NULL, 0, LoadLibraryW_addr, remote_addr, 0, NULL).
       - Creates a new thread in the game process that calls LoadLibraryW
         with the path to LIBERTEA.DLL.
       - LoadLibraryW loads the DLL, which triggers DllMain -> Phase 1-4.
    f) Wait for the thread to complete, then clean up (VirtualFreeEx to free
       the path string memory).

  METHOD 2: NtCreateThreadEx (STEALTHIER)
    - Uses the lower-level NT API instead of CreateRemoteThread.
    - Fewer hooks from anti-cheat on the NT path.
    - Same basic flow: allocate memory, write DLL path, create thread to
      call LoadLibrary.

  METHOD 3: Manual Mapping (MOST STEALTHY)
    - Instead of using LoadLibrary, the injector manually maps the DLL:
      a) Read the DLL file into a buffer.
      b) Parse the PE headers to determine section layout.
      c) Allocate memory in the game process for each section.
      d) Copy section data, apply relocations, resolve imports manually.
      e) Call the DLL's entry point (DllMain) directly.
    - No LoadLibrary call = nothing shows in the loaded modules list.
    - MUCH harder to detect because the DLL is not registered with the
      Windows loader.
    - This is the method LiberTea likely uses for maximum stealth.

H.4 POST-INJECTION — WHAT THE INJECTOR DOES AFTER
---------------------------------------------------
After successful injection:

  1. Wait for the DLL to initialize (may use an event/semaphore for
     synchronization).
  2. Check that the DLL has found the game and installed hooks
     (the DLL may communicate back via a named pipe, shared memory, or
     by setting a flag in a known location).
  3. If injection failed:
     - Log the error.
     - Clean up (free memory, close handles).
     - Display error to user.
  4. If injection succeeded:
     - The injector can either exit (the DLL runs independently inside the
       game process) or stay running as a "watchdog" to:
       a) Monitor the game process (is it still running?).
       b) Re-inject if the DLL is unloaded.
       c) Provide a communication channel for updates.

================================================================================
  APPENDIX I: THE PATTERN DATABASE — HOW 73 PATTERNS WORK
================================================================================

I.1 WHAT IS A PATTERN?
-----------------------
A pattern (also called a "signature" or "AOB signature") is a unique byte
sequence that identifies a specific location in compiled code.

Example pattern:
  Name: "GodMode_DamageFunction_Hook"
  Module: game.dll
  Signature: "48 89 5C 24 08 48 89 74 24 10 57 48 83 EC 30 48 8B F9"
  Hook type: FUNCTION_PROLOGUE
  Action: Write JMP to cheat handler at pattern match location.

I.2 HOW PATTERNS ARE FOUND
----------------------------
Patterns are created by:

  1. Finding the target function in the game's code (using IDA Pro, Ghidra,
     or a similar disassembler).
  2. Selecting a unique byte sequence at or near the function entry.
  3. Adding wildcards (??) for bytes that vary between game versions:
     - Relative addresses (call 0x12345 -> bytes depend on location).
     - RIP-relative addressing (lea rcx, [rip + offset]).
     - Constant values that might change (hardcoded limits, multipliers).
  4. Saving the pattern with metadata (name, module, hook type, action).

I.3 THE 73 ACTIVE PATTERNS
----------------------------
Distribution across DLLs:
  - game.dll: 67 patterns (91.8%).
  - winhttp.dll: 2 patterns (2.7%).
  - bcrypt.dll: 2 patterns (2.7%).
  - game_current.dll: 2 patterns (2.7%).

Distribution by hook type:
  - CODE_PATCH: 30 patterns (41.1%).
  - NOP_PATCH: 27 patterns (37.0%).
  - FUNCTION_PROLOGUE: 5 patterns (6.8%).
  - POINTER_RESOLVE: 5 patterns (6.8%).
  - FUNCTION_RETURN: 4 patterns (5.5%).
  - CONDITIONAL_INVERT: 2 patterns (2.7%).

I.4 PATTERN MATCHING ALGORITHM
-------------------------------
The scanner works as follows:

  1. Get the module's base address and size.
     - Module base: from GetModuleHandle or PEB traversal.
     - Size: from PE headers (SizeOfImage).

  2. For each pattern in the database:
     a) Convert the signature string to bytes with a mask:
        - "48 89 ?? 24" -> bytes = [0x48, 0x89, 0x00, 0x24], mask = [1,1,0,1].
     b) Scan the module memory:
        for (i = 0; i < module_size - pattern_len; i++):
          match = true;
          for (j = 0; j < pattern_len; j++):
            if (mask[j] && memory[i+j] != bytes[j]):
              match = false;
              break;
          if (match):
            found_pattern(i);
            break;
     c) If found, return: module_base + i (the address of the match).
     d) If not found, log: "Pattern X not found."

  3. Complexity: O(module_size * pattern_len) per pattern in worst case.
     - With 73 patterns and a ~50 MB game.dll, this is efficient enough
       (runs in under a second on modern CPUs).

  4. Pattern validation:
     - After finding a pattern, the cheat may verify it by checking
       surrounding bytes (expected instructions before/after).
     - This prevents false matches.

I.5 WHY 404 PATTERNS TESTED BUT ONLY 73 USED
-----------------------------------------------
The cheat tests 404 patterns but only 73 produce matches in the current
game version because:

  - Some patterns are for ALTERNATIVE VERSIONS of the game.
  - Some are FALLBACKS if the primary pattern for a feature fails.
  - Some are for OPTIONAL features that only work on certain game builds.
  - Some are LEGACY patterns from older versions (kept for backward
    compatibility attempts).
  - The "73 found" count is the SUCCESSFUL match count.

I.6 THE "GAME MAY HAVE UPDATED" WARNING
------------------------------------------
If fewer patterns match than expected:

  1. The cheat compares found count to expected count.
  2. If significantly fewer: display warning via ImGui or MessageBox.
  3. The warning tells the user:
     - The cheat may not work properly on this game version.
     - They should check for a cheat update.
  4. The cheat may disable certain features that depend on missing patterns
     rather than crashing.

================================================================================
  APPENDIX J: TROUBLESHOOTING — COMMON PROBLEMS AND ANSWERS
================================================================================

J.1 "THE CHEAT WON'T INJECT."
------------------------------
Possible causes:
  - GameGuard driver is still active (didn't get deleted/blocked properly).
  - The injector doesn't have administrator privileges (SeDebugPrivilege
    requires running as admin).
  - Anti-virus is blocking the injector (add exception or disable during use).
  - The game process is protected by a different anti-cheat (verify it's
    Helldivers 2).
  - Wrong game version — the injector may check the game's version before
    attempting injection.
  - DLL file is missing or corrupted (check file size: should be ~732 KB).

J.2 "THE MENU DOESN'T APPEAR."
-------------------------------
Possible causes:
  - The overlay hook (wglSwapIntervalEXT) failed to install.
  - ImGui failed to initialize (missing font, OpenGL context issue).
  - The game is running in fullscreen exclusive mode, preventing overlay
    windows. Try borderless windowed mode.
  - Another overlay (Steam overlay, Discord overlay, GeForce Experience)
    is conflicting. Disable other overlays.
  - The toggle key (INSERT/HOME) is captured by another program.
  - The overlay thread crashed silently.

J.3 "HOOKS ARE NOT WORKING (GOD MODE DOESN'T WORK, ETC.)."
-----------------------------------------------------------
Possible causes:
  - Game has updated and patterns no longer match. Check the pattern scanner
    output for warnings.
  - The cheat is running but hooks are being DETECTED and RESTORED by
    GameGuard or the game's own integrity checks.
  - Feature is not enabled correctly (check the checkbox toggle in the menu).
  - Another cheat or mod is conflicting with the hooks.

J.4 "SUPER CREDIT FARMING NOT GENERATING SC."
-----------------------------------------------
Possible causes:
  - No capture file: C:\libertea_replay_cap.json doesn't exist. The cheat
    needs to capture a legitimate Mission/end request first.
  - The capture file is corrupted or empty.
  - The HTTP library (WinHTTP/WinINet/libcurl) is blocked or not working.
  - The server is rate-limiting or has changed its API endpoint.
  - The hash table NOP isn't working (duplicate detection is rejecting
    replays).
  - MIDSWAP is generating invalid mission IDs.
  - Network connectivity issues (firewall, VPN, proxy blocking the API
    endpoint).
  - Authentication tokens have expired (the game session may need to be
    refreshed).

J.5 "THE GAME CRASHES WHEN USING THE CHEAT."
----------------------------------------------
Possible causes:
  - A hook is installed at the wrong location (pattern matched incorrectly
    due to game update).
  - A hook is corrupting the game's memory (wrong bytes written, wrong NOP
    length, stack corruption).
  - The overlay is causing a rendering conflict (OpenGL context issues).
  - The cheat and another program are competing for the same resources.
  - The VEH crash handler failed to recover from a minor exception, causing
    a cascade failure.
  - Memory exhaustion (too many allocations).
  - A syscall is being blocked or returning unexpected results on this
    Windows version.

J.6 "ANTI-VIRUS FLAGS THE CHEAT."
-----------------------------------
Expected. The cheat exhibits behaviors strongly associated with malware:
  - Process injection.
  - Memory modification.
  - DLL loading from non-standard paths.
  - Direct syscall usage.
  - Packed/compressed code.

This is why the cheat is packed — to avoid signature detection. Even with
packing, heuristic (behavioral) detection may still flag it.

J.7 "THE CHEAT WORKED YESTERDAY BUT NOT TODAY."
------------------------------------------------
Most likely cause: Helldivers 2 received an automatic update (via Steam).
Game updates can:
  - Change memory offsets and patterns.
  - Modify the code that hooks target.
  - Update GameGuard.
  - Change API endpoints or request formats.

The cheat must be updated by its authors to match the new game version.

J.8 "MY FRIEND'S CHEAT WORKS BUT MINE DOESN'T — SAME VERSION."
----------------------------------------------------------------
Possible causes:
  - Different Windows versions (SSN table is different — syscall numbers
    vary between Windows 10, Windows 11, and different builds).
  - Different hardware (GameGuard may behave differently on different CPUs,
    especially with virtualization-based security).
  - Different game launch options (Steam launch parameters, graphics settings).
  - Different network configuration (proxy, VPN, firewall rules).
